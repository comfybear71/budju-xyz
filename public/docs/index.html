1. Project Overview

BUDJU is a Solana-based meme coin with an integrated DCA (Dollar Cost Averaging) Trading Bot. It is a full-stack web application built with a React/TypeScript frontend and Python/Node.js serverless backend, deployed on Vercel.

Token Info

  • Name: BUDJU Coin
  • Tagline: "it's a lifestyle"
  • Total Supply: 1,000,000,000 tokens
  • Created: January 31, 2025
  • Blockchain: Solana (mainnet-beta)
  • Decimals: 6

Application Info

  • Type: Single Page Application (SPA)
  • Frontend: React 19 + TypeScript 5.7
  • Backend: Vercel Serverless Functions
  • Database: MongoDB
  • Hosting: Vercel
  • CI/CD: GitHub Actions

Data Flow Patterns

Wallet Balance Flow

Browser → /api/rpc → Helius/Solana RPC → getBalance, getTokenAccountsByOwner → SOL + BUDJU + USDC balances returned to WalletContext.

Trading Flow

TradePanel → /api/proxy → Swyftx API → Order placed (market/limit/stop) → Result recorded in MongoDB via /api/trade.

Price Feed Flow

/api/proxy → CoinGecko → USD + AUD prices → Live AUD→USD rate derived from BTC reference → Cached 25 seconds client-side.

Deposit Flow

User sends USDC on-chain → depositService.ts builds SPL transfer → Phantom signs → tx confirmed → /api/user-deposit records shares in MongoDB.

NAV (Net Asset Value) Model

NAV = Total Pool Value / Total Shares
Shares Issued = Deposit Amount / Current NAV
User Value = User Shares x Current NAV
User Allocation % = (User Shares / Total Shares) x 100

Example:
  Pool initialized: $1000 = 1000 shares at NAV $1.00
  User A deposits $100 -> 100 shares at NAV $1.00
  Pool grows to $1200 (1100 total shares)
  New NAV = $1200/1100 = $1.0909
  User A value = 100 x $1.0909 = $109.09 (9.1% gain)

Transaction Types Supported

  • Legacy Transactions: Classic Solana transaction format
  • Versioned Transactions (V0): Modern format with address lookup tables
  • signAndSendTransaction: Combined signing + sending (preferred)
  • signTransaction + sendRawTransaction: Fallback for wallets without combined method

Connection Persistence

Wallet connections are persisted in localStorage with three keys:

budjuWalletAddress  - The connected wallet's public key
budjuWalletName     - "phantom" | "solflare" | "jupiter" | "other"
budjuWalletConnected - "true"

On page reload, the app checks localStorage, waits for the wallet provider to inject (up to 2 seconds), then auto-reconnects. On mobile in-app browsers, it eagerly reconnects even if isConnected is false.

Balance Tracking

The walletService polls balances every 30 seconds via the RPC proxy:

  • SOL: via getBalance
  • BUDJU: via getTokenAccountsByOwner (mint-specific)
  • USDC: via getTokenAccountsByOwner (always included)
  • Additional custom tokens configurable

9. Trading Bot Architecture (AutoTrader)

Access Requirement Users must hold 10,000,000 BUDJU tokens to unlock the trading bot.

Tier System

TierNameDefault CoinsDeviation RangeAllocation Range
Tier 1Blue ChipsBTC, ETH, SOL, BNB, XRP1-15%1-25%
Tier 2AltsUser-assigned2-20%1-20%
Tier 3SpeculativeUser-assigned3-30%1-15%

Trading Logic

  • Sets buy/sell targets when a tier starts: buy = price * (1 - deviation%), sell = price * (1 + deviation%)
  • After BUY: buy target moves down, sell target stays
  • After SELL: sell target moves up, buy target stays
  • 24-hour cooldown per coin after each trade
  • Always keeps $100 USDC minimum reserve
  • Sells only 83.3% of allocation (accumulation bias)
  • Price checks every 3 minutes

Order Types via Swyftx

Type IDNameTrigger Condition
1Market BuyImmediate execution
2Market SellImmediate execution
3Limit BuyPrice drops to target
4Limit SellPrice rises to target
5Stop BuyPrice rises above target
6Stop SellPrice drops below target

Device Ownership

Only one device can run the auto-trader at a time. The system uses a heartbeat mechanism:

  • Each device gets a unique deviceId stored in localStorage
  • Active device sends heartbeat every 3 minutes
  • Heartbeat is stale after 5 minutes → other devices can reclaim ownership
  • Ownership verification every 9 minutes (every 3rd check)

12. Security & Authentication

Server-Side Security

API Key Protection

All API keys (Swyftx, Helius, Jupiter, Telegram, MongoDB) are stored as environment variables on Vercel and never exposed to the client browser. All external API calls are proxied through serverless functions.

RPC Method Allowlist

The /api/rpc proxy only allows 18 read-only Solana RPC methods. Write methods like sendTransaction are blocked to prevent abuse.

Allowed: getAccountInfo, getBalance,
getBlock, getBlockHeight,
getLatestBlockhash, getTransaction,
getTokenAccountsByOwner, ...

Swyftx Endpoint Allowlist

The proxy only forwards requests to approved Swyftx endpoints matching specific URL patterns. Unauthorized endpoints are rejected with 403.

Admin Authorization

Admin-only endpoints check the wallet address against a comma-separated list in the ADMIN_WALLETS environment variable. Non-admin wallets receive 403 errors.

Wallet Signature Verification

The Python backend uses PyNaCl for Ed25519 signature verification, validating that requests come from the claimed wallet owner. Address validation is done via base58 decoding.

CORS Configuration

All API endpoints set Access-Control-Allow-Origin: * with support for OPTIONS preflight requests.

15. Warnings & Security Concerns

RESOLVED: CORS Wildcard on All API Endpoints All API endpoints now restrict Access-Control-Allow-Origin to https://budju.xyz and https://www.budju.xyz only. Localhost is allowed for development. The Vary: Origin header is set for proper caching.
RESOLVED: Admin Authorization Now Uses Ed25519 Signatures All admin write endpoints now require cryptographic Ed25519 signature verification. Admin requests must include adminSignature and adminMessage (with timestamp for replay protection). The server verifies the signature matches the admin wallet's public key.
RESOLVED: Rate Limiting Added to All API Endpoints In-memory per-IP rate limiting has been added to all serverless functions. Limits: 30 req/min (proxy), 60 req/min (RPC), 20 req/min (Jupiter), 30 reads/10 writes per min (Python API). Returns HTTP 429 when exceeded.
RESOLVED: Trading PIN Removed The trade PIN (budju_trade_pin) localStorage code has been removed. This dead code was a security risk and has been cleaned up.
WARNING: Console Logging of Sensitive Data Multiple files contain console.log() statements that output RPC responses, swap transaction data, wallet addresses, and trade details. This data is visible in the browser DevTools to any user. Consider removing or reducing verbose logging in production builds.
WARNING: Deposit Service Simulated Function The executeDeposit function in useTrading.ts returns a mock transaction ID (SimulatedDepositTx...). If this is called from a production path, users may believe a deposit succeeded when it did not. Verify this code path is not reachable in production or replace with real implementation.
WARNING: Hardcoded Fallback AUD/USD Rate The AUD_TO_USD variable defaults to 0.708. If CoinGecko fails to return data, all AUD-denominated calculations (trigger prices, portfolio values) will use a potentially stale rate. Consider adding age tracking and alerts when the rate hasn't been updated.
WARNING: No Input Validation on Trade Amounts While there's a minimum order check ($7 market, $50 limit), there's no maximum check or balance verification before sending the order to Swyftx. Consider adding server-side amount validation against available balances.
INFO: No Automated Tests The project has no unit tests, integration tests, or end-to-end tests. Given the financial nature of the application (trading, deposits, withdrawals), this is a significant risk. Critical paths like deposit recording, share calculation, and trade execution should have comprehensive test coverage.
INFO: CI/CD Only Deploys to Vercel (No Testing Stage) The GitHub Actions workflow runs npm ci, npm run build, and deploys directly. There are no test, lint, or type-check steps in the pipeline. A failing build is caught, but runtime bugs will reach production.
INFO: Single Database Region MongoDB is accessed directly from serverless functions. Consider using MongoDB Atlas with connection pooling and read replicas for better performance and redundancy.
INFO: Telegram Bot Token Exposure Risk The Telegram bot token is used in the serverless function. While it's server-side, if the function error-logs the token or it appears in stack traces, it could be exposed. Ensure error logging doesn't include env vars.
