Data Flow Patterns

  • Wallet Balance: Browser → /api/rpc → Helius/Solana → getBalance + getTokenAccountsByOwner
  • Trading: TradePanel → /api/proxy → Swyftx API → Order placed → MongoDB record via /api/trade
  • Price Feed: /api/proxy → CoinGecko → USD+AUD prices → AUD/USD rate derived from BTC → 25s cache
  • Deposit: User sends USDC on-chain → depositService builds SPL transfer → Wallet signs → /api/user-deposit records shares
  • Swap: Quote via /api/jupiter (GET) → Swap tx via /api/jupiter (POST) → Wallet signs versioned tx

Trading Bot Details (Core Feature)

The trading bot requires holding 10,000,000 BUDJU tokens. It provides:

  • TradePanel: Manual market/limit/stop orders via Swyftx
  • AutoTraderView: 3-tier automated DCA engine
  • AdminAutoTradeView: Admin bot management panel
  • PendingOrdersView: Queued orders from Swyftx
  • PortfolioChart: Real-time asset allocation donut chart
  • Leaderboard: User rankings by portfolio value
  • ActivityLog: Trade history with timestamps
  • TransactionHistory: All deposits, trades, withdrawals
  • HoldingsList: Individual asset positions with P&L

7. Database Schema (MongoDB)

Collection: users

{
  walletAddress: String (unique index),
  shares: Float,              // User's share count in pool
  allocation: Float,          // Allocation % in pool
  totalDeposited: Float,      // Cumulative deposits (USD)
  totalWithdrawn: Float,      // Cumulative withdrawals
  holdings: Object,           // Asset holdings {BTC: 0.5, ETH: 2.0}
  joinedDate: DateTime,
  lastLogin: DateTime,
  isActive: Boolean
}

Collection: deposits

{
  userId: String (wallet address),
  amount: Float,              // Deposit amount (USD/USDC)
  currency: "USDC" | "AUD",
  txHash: String (unique),    // Transaction hash (dedup key)
  shares: Float,              // Shares issued at this deposit
  nav: Float,                 // NAV at time of deposit
  timestamp: DateTime,
  status: "completed",
  source: "client_sync" | "swyftx_sync"
}

Collection: trades

{
  coin: String,               // Asset code (BTC, ETH, etc.)
  type: "buy" | "sell",
  amount: Float,
  price: Float,
  timestamp: DateTime,
  swyftxId: String (unique),  // Swyftx order UUID
  userAllocations: Object,    // {wallet: %allocation}
  source: String
}

Other Collections

  • pool_state: Single document (_id: "pool") with totalShares, initialized, recalibrated dates
  • trader_state: Single document (_id: "admin_state") with pending orders, tier configs, bot state, cooldowns, trade log
  • withdrawals: userId, amount, currency, timestamp, status

NAV (Net Asset Value) Model

NAV = Total Pool Value / Total Shares
Shares Issued = Deposit Amount / Current NAV
User Value = User Shares x Current NAV
User Allocation % = (User Shares / Total Shares) x 100

9. Trading Bot Architecture (AutoTrader)

Tier System

TierNameDefault CoinsDeviationAllocation
1Blue ChipsBTC, ETH, SOL, BNB, XRP1-15%1-25%
2AltsUser-assigned2-20%1-20%
3SpeculativeUser-assigned3-30%1-15%

Core Trading Logic

  1. Set buy target = price x (1 - deviation%), sell target = price x (1 + deviation%)
  2. Monitor prices every 3 minutes
  3. If price ≤ buy target → execute market BUY, then move buy target DOWN
  4. If price ≥ sell target → execute market SELL, then move sell target UP
  5. Apply 24-hour cooldown per coin after each trade
  6. Always maintain $100 USDC minimum reserve
  7. Sell only 83.3% of allocation (accumulation bias)

Order Types via Swyftx

IDTypeTrigger
1Market BuyImmediate
2Market SellImmediate
3Limit BuyPrice drops to target
4Limit SellPrice rises to target
5Stop BuyPrice rises above target
6Stop SellPrice drops below target

Device Ownership

Single-device execution: unique deviceId in localStorage, heartbeat every 3 minutes, stale after 5 minutes. Other devices can reclaim ownership after staleness.

12. Security & Authentication

Implemented Security Measures

  • Server-Side Secret Storage: All API keys (Swyftx, Helius, Jupiter, Telegram, MongoDB) stored as Vercel env vars, never exposed to browser
  • RPC Method Allowlist: /api/rpc only allows 18 read-only Solana methods; write methods (sendTransaction) are blocked
  • Swyftx Endpoint Allowlist: Proxy only forwards to approved URL patterns; unauthorized endpoints return 403
  • Admin Authorization: Admin-only endpoints check wallet against ADMIN_WALLETS env var
  • Wallet Signature Verification: Ed25519 verification available via PyNaCl (optional usage)
  • Address Validation: base58 decoding for Solana address validation
  • CORS Preflight: OPTIONS request handling on all endpoints

15. Warnings & Security Concerns

CRITICAL #1: CORS Wildcard on All API Endpoints All serverless functions set Access-Control-Allow-Origin: *. Any website can make requests to your API. While API keys are server-side, MongoDB endpoints (/api/deposit, /api/trade, /api/state) are callable from any origin. Fix: Restrict CORS to your production domain.
CRITICAL #2: Admin Authorization Is Wallet-Address-Only Admin endpoints verify wallet address against ADMIN_WALLETS env var, but this is a plain string in the request body — not cryptographically proven. An attacker who knows an admin address could forge admin requests. Fix: Require Ed25519 signature verification on all admin write operations.
CRITICAL #3: No Rate Limiting No rate limiting on any serverless function. Endpoints like /api/user/register, /api/user-deposit, /api/trade can be flooded. Fix: Add Vercel Edge Middleware rate limiting or use Upstash Redis.
RESOLVED #4: Trading PIN Removed The trade PIN (budju_trade_pin) localStorage code has been removed. This dead code was a security risk and has been cleaned up.
WARNING #5: Verbose Console Logging in Production Multiple files log RPC responses, transaction data, wallet addresses, and trade details to console.log(). Visible in browser DevTools. Fix: Strip console.log in production builds or use a logging library with log levels.
WARNING #6: Simulated Deposit Function The executeDeposit function in useTrading.ts returns a mock transaction ID ("SimulatedDepositTx..."). If reachable in production, users may think a deposit succeeded when it didn't. Fix: Replace with real implementation or remove the code path.
WARNING #7: Hardcoded Fallback AUD/USD Rate (0.708) If CoinGecko fails, all AUD calculations use a static 0.708 rate. Trigger prices could be significantly off. Fix: Add staleness tracking and alert when rate hasn't updated in >5 minutes.
WARNING #8: No Maximum Trade Amount Validation While minimum orders are checked ($7 market, $50 limit), there's no maximum check or server-side balance verification before sending to Swyftx. Fix: Add server-side balance checks.
INFO #9: No Automated Tests No unit, integration, or E2E tests exist. For a financial application handling trades and deposits, this is a significant risk. Fix: Add Vitest for unit tests and Playwright for E2E tests, especially for NAV calculations and trade execution.
INFO #10: CI Pipeline Has No Test Stage GitHub Actions runs build + deploy directly. No lint, type-check, or test verification. Runtime bugs reach production undetected. Fix: Add npm run lint and npm run test stages before deployment.
INFO #11: Single Database Without Redundancy MongoDB is accessed directly from serverless functions without connection pooling. Consider MongoDB Atlas with read replicas for better reliability.
INFO #12: Telegram Bot Token Exposure Risk If error logging includes environment variables or stack traces, the bot token could leak. Ensure error handlers sanitize output.
